# DurableStack Hosted App Platform

## Revised Scope and Product Definition

Version: Draft 2
Product: DurableStack Hosted App Platform
Related Product: DurableStack Runtime and Hosted Telemetry System
Primary Runtime Stack: ASP.NET Core / .NET 9
Frontend Framework: Tailwind CSS (mobile-first)
Primary UI Pattern: Real-time operational control plane

---

# 1. Product Overview

The DurableStack Hosted App Platform is the centralized operational and management interface for DurableStack tenants, projects, workers, jobs, and execution telemetry.

The purpose of the platform is to provide customers with:

* Real-time operational visibility
* Distributed worker monitoring
* Durable job execution analytics
* Failure analysis and retry visibility
* Multi-project dashboards and reporting
* Team and tenant management
* Secure credential management
* Alerts, exports, and integrations

The hosted platform is a major component of the DurableStack monetization strategy.

The platform should provide a significantly better operational experience than raw OpenTelemetry integrations or customer-built dashboards, making the hosted service the preferred interface for monitoring and managing DurableStack environments.

---

# 2. Product Positioning

DurableStack runtime solves:

> “How do I reliably execute durable background jobs?”

The hosted app platform solves:

> “How do I monitor, understand, manage, and operate those jobs in production?”

The hosted platform should feel comparable in quality and usability to modern operational SaaS platforms such as:

* Datadog
* Sentry
* Grafana Cloud
* Vercel
* PlanetScale
* GitHub Actions dashboards
* Azure Application Insights

The experience should prioritize:

* clarity
* operational visibility
* ease of use
* mobile usability
* fast navigation
* actionable information
* minimal configuration friction

# 3. Core Product Principles

## 3.1 Project-Aware, Not Project-Trapped

Users must be able to:

* view data across all projects
* filter down to individual projects
* compare multiple projects
* build dashboards with mixed project scopes

The system should not force hard project isolation in the UI experience.

Example:

A dashboard may contain:

* Widget A → Runs for Project A
* Widget B → Runs for Project B
* Widget C → Failures across all projects

---

## 3.2 Real-Time Operational Visibility

The long-term dashboard experience is intended to be near-real-time.

For MVP, prioritize authentication, onboarding, and core operational pages first.

Near-real-time auto-refresh is a post-onboarding phase.

Reports do not require auto-refresh.

---

## 3.3 Mobile-First Design

The application must be designed mobile-first from the beginning.

Desktop support is important, but the system should remain highly usable on phones and tablets.

---

## 3.4 Operational Clarity

The application should prioritize:

* status visibility
* trend visibility
* failures
* retries
* worker health
* actionable operational insights

The platform should avoid becoming cluttered or overly developer-centric.

# 9. Real-Time Data Refresh Requirements

Phase note:

* implement after auth + onboarding foundation is complete

Dashboard widgets should auto-refresh approximately every 15 seconds.

API endpoints must support incremental refresh.

Required API capabilities:

* initial dataset retrieval
* changed-since timestamp querying
* server timestamps in responses
* filtering support
* lightweight polling

Suggested API pattern:

```http
GET /api/dashboard/widgets/{id}?sinceUtc=2026-05-23T20:00:00Z
```

Reports do not require auto-refresh behavior.

* create projects/environments

---

# 27. Relationship to DurableStack Runtime

The hosted platform must never interfere with runtime correctness.

If hosted telemetry upload fails:

* jobs must continue executing
* retries must continue functioning
* scheduling must continue functioning
* worker execution semantics must remain intact

The hosted layer is operational and observational, not execution-critical.

This aligns with the DurableStack hosted sync design goals already established in the runtime build documentation.
